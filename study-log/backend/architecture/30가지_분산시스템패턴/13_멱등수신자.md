# 목적
클라이언트 요청을 고유하게 식별해 클라이언트가 재시도하더라도 중복 요청을 무시할 수 있게 한다.

# 문제
서버가 요청을 처리하고 죽었다면 클라이언트가 재시도할 때 서버는 중복 요청을 받게 된다

# 기본 동작
- 서버는 **클라이언트마다 고유 ID를 부여**하고, 클라이언트는 요청 전에 등록한다.
- 서버는 **세션을 생성**해 클라이언트 요청의 응답을 저장하고, 비활성 세션은 제거한다.
- 클라이언트 등록과 응답 저장은 **합의 알고리즘으로 복제**되므로, 리더가 바뀌어도 유지된다.
- 클라이언트가 **요청 번호를 포함**해 재시도하면 서버는 이전 응답을 저장해 **같은 요청에 동일한 응답 반환**
    → 중복 실행 없이 안정적인 처리 가능

# 주의사항
- 중복 요청 제거는 연결 실패 후 재시도 상황에서만 적용되며, 클라이언트가 재시작된 경우에는 새로 등록되므로 중복 제거 대상이 아니다
- 클라이언트 세션은 영구 저장되지 않으며 TTL(Time-to-live)이 설정되고, 클라이언트는 주기적으로 하트비트를 전송
- 서버는 TTL 내 하트비트가 없으면 세션을 제거하고 주기적으로 만료된 세션을 정리한다.

# 요청 처리 보장 방식
- 클라이언트 실패 및 재시도 방식에 따라 요청 처리 보장이 달라진다
  - 최대 한 번 (At-Most-Once): 클라이언트가 재시도하지 않음, 요청이 처리됐을 수도, 안 됐을 수도 있음 (0 또는 1회)
  - 최소 한 번 (At-Least-Once): 클라이언트가 재시도함, 요청이 여러 번 처리될 수 있음 (중복 가능)
  - 정확히 한 번 (Exactly-Once): 재시도하더라도 한 번만 처리 보장,  이를 위해 멱등 수신자(idempotent receiver) 필요
- 중복 처리 방지(멱등성)가 정확히 한 번 실행의 핵심 조건이다.
