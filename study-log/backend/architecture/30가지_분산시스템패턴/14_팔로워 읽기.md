# 목적
팔로워에서 읽기 요청을 처리해 처리량과 지연 시간을 개선한다

# 문제 
- 리더 팔로워(Leader and Followers) 패턴을 사용할 때 리더가 너무 많은 요청을 받으면 과부하가 걸릴 수 있다
- 클라이언트가 원격 데이터센터에 있는 다중 데이터센터 구성에서는 리더로 요청을 보낼 때 추가적인 지 연 시간이 발생한다

# 해결
- 쓰기 요청은 일관성을 위해 반드시 리더 노드로 전송해야 한다.
- 읽기 요청은 성능을 위해 가까운 팔로워에서 처리 가능하다.
  - 약간 오래된 데이터를 허용할 수 있는 경우에만 적합
  - 복제 지연(replication lag) 때문에 최신이 아닌 오래된 값(stale data)을 반환할 수 있기 때문
 
# 팔로워 선택 및 연결
- 클러스터 노드는 위치(리전) 정보 같은 메타데이터를 관리하며, 클라이언트는 이를 기반으로 가장 가까운 복제 서버를 선택할 수 있다.
- 하트비트로 지연 시간(latency)을 측정하고 지연 시간이 가장 낮은(이동 평균) 팔로워를 선택한다.
- 노드 간 통신은 단일 소켓 연결 + 하트비트 유지를 통해 지연 측정과 관리가 용이하다.

# 일관성 문제 - 버전 기반 검증으로 “자신이 쓴 값 읽기(Read-your-writes)” 보장이 필요함
- 팔로워는 네트워크 단절이나 디스크 지연으로 리더보다 뒤처질 수 있음
  - 일정 시간 이상 지연되면 요청 처리를 중단하거나 선택 대상에서 제외
- 클라이언트가 쓰기 후 바로 읽기 시 아직 복제 안 된 팔로워에서 오래된 값 반환 가능
  - Read-Your-Writes 일관성
    - 쓰기 시 버전 스탬프(버전 번호) 함께 저장, 클라이언트는 응답으로 버전 받음
    - 읽기 시 클라이언트가 버전 스탬프를 함께 전달, 팔로워는 자신의 데이터 버전 ≥ 요청 버전 → 즉시 응답 아니면 최신 데이터까지 기다림
   
# 강한 일관성 = 리더 읽기 + 리더 유효성 검증
- 최신 데이터가 반드시 필요한 경우(선형성 읽기)에는 → 읽기 요청을 리더에게 보내야 한다
- 리더는 응답 전에 자신이 여전히 리더인지 확인하는 절차가 필요함
- 일관성 코어(Consistent Core) 기반 시스템에서 일반적으로 사용하는 설계 방식
